## RAG란?

RAG(Retrieval-Augmented Generation)는 LLM이 답변을 생성하기 전에 **외부 지식 저장소에서 관련 정보를 검색해서 프롬프트에 끼워 넣는** 기법이다. LLM의 두 가지 근본 한계를 보완한다.

- **지식 컷오프**: 모델은 학습 시점 이후의 정보를 모른다
- **환각(hallucination)**: 모르는 것을 아는 척 지어낸다

## 기본 파이프라인

```
문서 → 청킹 → 임베딩 → 벡터DB 저장   (인덱싱, 사전 작업)
질문 → 임베딩 → 유사도 검색 → 상위 k개 문서 + 질문 → LLM → 답변   (질의 시점)
```

### 1. 청킹 (Chunking)

문서를 검색 가능한 단위로 자르는 과정. 너무 크면 관련 없는 내용이 섞이고, 너무 작으면 맥락이 끊긴다. 보통 300~1000 토큰에 청크 간 일부 겹침(overlap)을 둔다.

### 2. 임베딩 (Embedding)

텍스트를 의미를 담은 숫자 벡터로 변환. 의미가 비슷한 문장은 벡터 공간에서 가까운 위치에 놓인다. OpenAI `text-embedding-3`, 한국어 특화 모델(KURE 등), 로컬은 Ollama의 `nomic-embed-text` 등이 쓰인다.

### 3. 벡터 DB

임베딩을 저장하고 유사도 검색을 해주는 저장소.

| 이름 | 특징 |
|---|---|
| Chroma | 가볍고 로컬 개발에 적합 |
| pgvector | PostgreSQL 확장 — 기존 DB에 얹기 좋음 |
| Pinecone | 관리형 서비스, 스케일 편함 |
| SQLite-vec | 임베디드, 소규모 프로젝트용 |

### 4. 생성

검색된 청크를 "다음 자료를 참고해서 답하라"는 프롬프트와 함께 LLM에 전달한다. 자료에 없으면 모른다고 답하게 지시하는 것이 환각 방지의 핵심이다.

## 품질을 올리는 기법들

- **하이브리드 검색**: 벡터 검색 + 키워드 검색(BM25)을 함께 사용
- **리랭킹(Reranking)**: 1차 검색 결과를 별도 모델로 다시 정렬
- **쿼리 변환**: 사용자의 질문을 검색에 유리한 형태로 재작성

> 개인 프로젝트 기준: Chroma 또는 pgvector + Ollama 임베딩 모델 조합이면 비용 0원으로 RAG를 실험할 수 있다.
