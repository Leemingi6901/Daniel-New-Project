## 왜 로컬 LLM인가

- **비용 0원**: API 호출량 걱정 없이 밤새 배치를 돌릴 수 있다
- **프라이버시**: 데이터가 내 기기를 떠나지 않는다
- **호출 제한 없음**: rate limit에서 자유롭다

반대급부는 명확하다: **모델 성능이 하드웨어에 묶인다.** 16GB RAM 기준 8B급 모델이 실용 상한선이다.

## Ollama 기본기

```bash
brew install ollama          # 설치
ollama pull qwen3:8b         # 모델 다운로드
ollama run qwen3:8b          # 대화 테스트
curl http://localhost:11434/api/generate -d '{"model":"qwen3:8b","prompt":"..."}' # API
```

Ollama는 `localhost:11434`에 OpenAI 호환 API도 제공하므로, 대부분의 LLM 프레임워크에 그대로 연결된다.

## 모델 선택 기준 (16GB RAM)

| 용도 | 추천 | 이유 |
|---|---|---|
| 에이전트/도구 호출 | qwen3:8b | 8B급에서 tool calling 지원이 가장 안정적 |
| 텍스트 요약·분류 배치 | qwen3:8b, gemma2:9b | 정형화된 작업은 8B로 충분 |
| 임베딩 (RAG용) | nomic-embed-text | 가볍고 품질 무난 |

## 직접 겪은 8B 모델의 현실

로컬 8B 모델을 에이전트로 운영하면서 배운 것:

1. **정형화된 배치 작업은 합격점** — 요약, 분류, 태깅, JSON 추출은 프롬프트만 잘 짜면 안정적이다.
2. **다단계 판단은 무너진다** — "상황을 보고 A 또는 B를 해라" 같은 분기가 겹치면 엉뚱한 도구를 실행한다. 실제로 인사말에 파일을 덮어쓴 사례가 있었다.
3. **세션이 길어지면 앵무새가 된다** — 이전 대화의 자기 답변을 그대로 반복하는 현상. 세션을 자주 초기화하는 것이 답이다.

## 실전 팁

- 한 호출에는 한 가지 작업만 (요약 따로, 분류 따로)
- 출력 형식은 JSON 스키마로 강제하고 파싱 실패 시 재시도
- temperature 0.1~0.3으로 낮게
- 품질이 중요한 최종 출력만 API 모델을 섞는 **하이브리드 구성**이 정석
