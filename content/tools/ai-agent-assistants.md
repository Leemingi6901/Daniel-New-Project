## 에이전트 도구의 세 갈래

2026년 현재 개인이 쓸 수 있는 AI 에이전트 도구는 크게 세 부류로 나뉜다.

| 부류 | 대표 | 특징 |
|---|---|---|
| 코딩 에이전트 | Claude Code, Codex CLI | 저장소 안에서 코드를 읽고 쓰는 개발 특화 |
| 개인 비서 게이트웨이 | OpenClaw | 메신저(텔레그램 등)로 대화하는 상시 비서 |
| 워크플로 프레임워크 | LangGraph, CrewAI, MetaGPT | 코드로 에이전트 파이프라인을 직접 조립 |

## 코딩 에이전트

터미널/IDE에서 저장소를 통째로 이해하고 수정한다. 파일 편집, 테스트 실행, git 커밋까지 수행. `AGENTS.md`(또는 `CLAUDE.md`) 파일에 프로젝트 규칙을 적어두면 에이전트가 따르는 관례가 자리잡았다.

## 개인 비서 게이트웨이 (OpenClaw)

상시 실행되는 게이트웨이가 메신저와 LLM 사이를 중개한다. 구조상 특징:

- **채널**: 텔레그램/디스코드/슬랙 등을 봇으로 연결, 모르는 사람은 페어링 코드로 차단
- **작업 공간**: 에이전트마다 폴더를 배정 — 정체성(IDENTITY.md)과 규칙(AGENTS.md)을 파일로 관리
- **모델 유연성**: API 모델부터 Ollama 로컬 모델까지 교체 가능
- **멀티 에이전트**: 역할별 에이전트를 여러 개 두고 채널별로 라우팅 가능 — "AI 직원" 구조

## 프레임워크 계열

메시지로 대화하는 비서가 아니라, 정해진 파이프라인(기획→개발→리뷰 등)을 자동으로 도는 시스템을 코드로 만든다. MetaGPT/ChatDev는 아예 "소프트웨어 회사 시뮬레이션"을 표방한다.

## 선택 기준

- 코드 작업이 중심이면 → 코딩 에이전트
- "살아있는 비서"와 대화하고 싶으면 → 게이트웨이 계열
- 사람 개입 없는 완전 자동 파이프라인이 목표면 → 프레임워크 계열

> 셋은 배타적이지 않다. 게이트웨이 비서가 필요할 때 코딩 에이전트를 호출하는 조합이 실제로 가장 실용적이다.
